# PathRare Project Analysis

## 1. File & Directory Usage Breakdown

### Root Configuration & Documentation Files
- **`package.json`, `package-lock.json`**: Define npm dependencies and project metadata.
- **`tsconfig.json`, `tsconfig.tsbuildinfo`, `next-env.d.ts`**: TypeScript configuration and type declaration files.
- **`next.config.ts`**: Configuration for the Next.js framework.
- **`eslint.config.mjs`, `postcss.config.mjs`**: Linting (ESLint) and CSS processing (PostCSS) configuration files.
- **`Dockerfile`, `.dockerignore`, `cloudbuild.yaml`, `deploy.ps1`**: Containerization and deployment configuration for Google Cloud Run.
- **`.env.local`**: Local environment variables (API keys, Firebase config).
- **`README.md`, `AGENTS.md`, `DATABASE_SCHEMA.md`, `GOOGLE_CLOUD_DEPLOY.md`**: Project documentation, database schema, and deployment guides.
- **`credentials/gcp-service-account.json`**: Service account keys for GCP (Ensure this is in `.gitignore`).

### `/app` Directory (Next.js App Router)
This folder serves as the routing mechanism for your application.
- **`layout.tsx`, `page.tsx`, `globals.css`**: The main root layout, the landing page, and global CSS styles.
- **`/api/...`**: Your backend route handlers, separated by domains like `/diagnose`, `/auth`, `/documents`, `/ngo`, `/tasks`, `/stt`, `/tts`, `/reports`, etc. These handle requests from the client to interact with databases and AI models.
- **`/auth/...`**: Contains authentication pages and the `complete-profile` routing.
- **`/patient/...`**: The main patient portal containing routes like `/diagnose`, `/care`, `/life-assist`, `/community`, and their respective sub-pages.
- **`/ngo/...`**: The dashboard and tools for NGO users (`/cases`, `/impact`, `/patients`, `/volunteers`).
- **`/volunteer/...`**: The dashboard and tools for volunteers (`/chat`, `/open` cases).
- **`/share/...` & `/shared/...`**: Public or token-gated routes for securely sharing medical profiles or documents.

*Note: The `/ngo` and `/volunteer` folders currently contain component files (e.g., `NgoHeatMap.tsx`, `shared-header.tsx`) and context providers (`ngo-dashboard-context.tsx`). See Section 3 for restructuring advice.*

### `/components` Directory
- **`LanguagePicker.tsx`**: A reusable UI component for selecting languages. This directory is currently heavily underutilized.

### `/lib` Directory
This folder acts as a catch-all for utilities, logic, and configurations.
- **`firebase.ts`, `firebase-admin.ts`**: Client and server-side Firebase initialization and configuration.
- **Contexts (`auth-context.tsx`, `language-context.tsx`)**: React Context providers for global state management.
- **Hooks (`use-t.ts`, `use-task-presence.ts`)**: Custom React hooks.
- **Business Logic (`rag.ts`, `support-workflow.ts`, `task-chat.ts`, `ngo-support.ts`)**: Core functions interacting with databases, handling task allocations, or running AI inference.
- **`/i18n/...`**: Internationalization text dictionaries for different pages.

### `/data` Directory
- Contains clinical datasets used for diagnostic algorithms and RAG models. Includes Human Phenotype Ontology (`/hpo`), Orphanet XMLs and parsed embeddings (`/orphanet`), and generic guidelines (`/rag`).

### `/scripts` Directory
- Contains Python scripts (`build_hpo_index.py`, `embed_rag_chunks.py`, `parse_orphanet.py`) used offline to parse XMLs and generate the processed JSON files and embeddings stored in `/data`.

## 3. Architecture Evaluation & Recommendations

**Current State**: Your structure uses modern Next.js 14+ conventions (App Router) which is great. However, it suffers from **"colocation clutter"**. You are storing UI components and React contexts directly inside the `app/` routing directories and mixing all utilities/hooks/services into a single `lib/` directory.

### To achieve a highly professional, industry-level structure, adopt the following changes:

#### A. Keep `/app` Strictly for Routing
In the App Router paradigm, folders inside `/app` should primarily contain `page.tsx`, `layout.tsx`, `loading.tsx`, `error.tsx`, and `route.ts`. 
- **Action**: Move files like `NgoHeatMap.tsx`, `association-manager.tsx`, `shared-header.tsx`, `ngo-association-panel.tsx`, and context files out of `/app/ngo/` and `/app/volunteer/`.

#### B. Implement Feature-Driven Architecture (Domain-Driven Design)
Instead of grouping files by their technical type (all components together, all hooks together), group them by their business domain. Create a `/features` folder at the root.
Example structure:
```
/features
  /ngo
    /components (NgoHeatMap.tsx, AssociationManager.tsx)
    /hooks (useNgoData.ts)
    /services (api calls, ngo-support.ts)
    /context (NgoDashboardContext.tsx)
  /volunteer
    /components (SharedHeader.tsx, AssociationPanel.tsx)
    /context (VolunteerDashboardContext.tsx)
  /auth
    /components
    /context (AuthContext.tsx)
  /diagnostics
    /services (rag.ts)
```

#### C. Dedicated Root Folders for Shared Architecture
For things that are truly global and not tied to one feature:
- Create `/providers`: Move `language-context.tsx`, `auth-context.tsx` here. Wrap your app layout in these providers.
- Create `/hooks`: Move shared hooks like `use-t.ts` here.
- Create `/ui` (inside `/components`): For dumb, reusable UI elements like `Button.tsx`, `LanguagePicker.tsx`, `Modal.tsx`.
- Create `/services` or `/lib/api`: Move Firebase admin init and external API handlers here.

#### D. API Route Thinning
Right now, your `/app/api/.../route.ts` files likely contain a lot of business logic. A professional standard is the **Controller-Service pattern**. The `route.ts` should only parse the request, call a function in `/services` (e.g., `diagnosePatient(data)`), and return the response.

### Summary of the Ideal File Tree:
- `/app` (Routing only: layout, page, route)
- `/components` 
  - `/ui` (Shared generic components)
- `/features` (Domain specific logic, components, and contexts: auth, ngo, patient)
- `/hooks` (Global custom hooks)
- `/providers` (Global React Contexts)
- `/services` (Backend logic, Firebase admin, external APIs)
- `/lib` (Pure utility functions like date formatting, string parsing)
- `/types` (Global TypeScript definitions)
