1. WHAT IS PATHRARE?
────────────────────
PathRare is a multi-role rare disease support platform. It connects patients
with rare diseases to NGOs, volunteers, coordinators, and medical institutions
through a single web application.

Core problems it solves:
  - Diagnostic uncertainty: patients with rare diseases wait years for a diagnosis
  - Resource discovery: hard to find NGOs and support groups
  - Care coordination: fragmented day-to-day support and task management

Roles supported:
  - Patient / Caregiver
  - NGO (Non-Governmental Organisation)
  - Volunteer
  - Doctor
  - Care Coordinator
  - Hospital

4. DATA SOURCES — WHAT EACH IS USED FOR
================================================================================

── ORPHANET DATA ──────────────────────────────────────────────────────────────

Source: Orphanet (www.orpha.net) — global reference database for rare diseases

  alignments.json (1.9 MB, ~105,000 entries)
    Purpose: Map every Orphanet ORPHA code to its ICD-11 and OMIM codes
    Used in: /api/diagnose — enriching disease matches with clinical codes
             /api/living-brief — including codes in the Living Brief
    Structure: [{ orpha_code, name, icd_codes: [], omim: [] }]

  classifications.json (5 MB, ~199,000 entries)
    Purpose: Full Orphanet disease taxonomy with types
             (Disease, Clinical group, Category, Malformation syndrome, etc.)
    Used in: /api/diagnose — supplementing missing disease names
    Structure: [{ orpha_code, name, type }]

  index.json (generated, not committed)
    Purpose: Pre-computed Vertex AI embedding vectors for each disease
    Generated by: scripts/embed_symptoms.py → scripts/build_index.py
    Used in: /api/diagnose (embedding-based pipeline)

── HPO DATA ───────────────────────────────────────────────────────────────────

Source: Human Phenotype Ontology (hpo.jax.org) — version 2026-02-16

  hp.obo (10.7 MB)
    What: Ontology of 19,944 standardised clinical phenotype terms
    Contains: HP codes, term names, synonyms, definitions, relationships
    Used in:
      - Building hpoTextLookup (lowercase term name → HP code)
        so patient text can be scanned for HPO terms without calling Gemini
      - Translating HP codes back to human-readable names in results
      - Populating synonym variants for text matching

  phenotype.hpoa (35 MB, 282,728 rows)
    What: Evidence-based disease ↔ HPO phenotype annotation database
    Sources: 4,337 Orphanet diseases + 8,614 OMIM diseases + 47 DECIPHER
    Columns: database_id | disease_name | qualifier | hpo_id | reference |
             evidence | onset | frequency | sex | modifier | aspect | biocuration
    Filtered to: aspect="P" (phenotypic abnormality) + qualifier≠"NOT"
    Used in:
      - diseaseHPO map: disease_id → Set<HPO codes>
        e.g. "ORPHA:140944" → {HP:0002650, HP:0010816, HP:0025104, ...}
      - hpoToDisease map: HPO code → Set<disease IDs>
        e.g. "HP:0002650" → {"ORPHA:140944", "ORPHA:14328", ...}
      - Jaccard similarity scoring for evidence-based disease ranking

5. DIAGNOSTIC PIPELINE — DETAILED FLOW
================================================================================

Two pipeline variants exist:

── VARIANT A: Gemini + HPO Database (current advanced version) ─────────────────

  INPUT: File upload (PDF/image) OR typed symptom text

  Step 1 — OCR (if file)
    Tool: Google Cloud Vision API (DOCUMENT_TEXT_DETECTION)
    Auth: GCP service account → access token via google-auth-library
    Input: Base64-encoded image bytes
    Output: Plain text transcript of the medical report

  Step 2 — HPO Text Pre-matching (no AI cost)
    Tool: In-memory hpoTextLookup (built from hp.obo at startup)
    Logic: Scan patient text for known HPO term names/synonyms (≥8 chars)
    Output: Set of HP codes found in text
    Example: "scoliosis" → HP:0002650, "port-wine stain" → HP:0025104

  Step 3 — HPO Disease Scoring (Jaccard similarity)
    Input: HP codes from Step 2
    Logic: For each HP code, look up matching diseases in hpoToDisease map
           Score = intersection / union (Jaccard similarity)
    Output: Top 20 ORPHA diseases ranked by phenotype overlap
    Example: CLOVES syndrome (ORPHA:140944) — 3 matching HPO terms

  Step 4 — Gemini API Call (single call)
    Model: gemini-2.0-flash (auto-discovered via /v1/models endpoint)
           Falls back through all available models on rate limit (429)
    Prompt includes:
      - Patient text (max 2000 chars)
      - HPO pre-matched diseases (flagged as "[HPO PRE-MATCHED]")
      - 200 general Orphanet diseases
    Gemini returns:
      - symptoms: [{term, hpo_code}] — each symptom with HP code
      - matches: top 5 diseases with confidence 0-100 + reasoning
    Note: Compact single-line JSON format to prevent truncation
          JSON repair function handles any remaining truncation

  Step 5 — Post-call HPO Scoring (merge pre + Gemini HP codes)
    Merge preHPOCodes + Gemini's extracted HP codes (deduplicated)
    Re-run Jaccard scoring with combined set for richer coverage

  Step 6 — Merge LLM + HPO Results
    For diseases in BOTH lists:
      combined_score = 0.55 × gemini_confidence + 0.45 × hpo_jaccard
    For Gemini-only diseases:
      combined_score = 0.55 × gemini_confidence
    For HPO-only diseases:
      combined_score = 0.45 × precision (matched_hpo / total_patient_hpo)
    Sort by combined_score, return top 5

  Step 7 — Enrich with Codes
    Look up each match in alignments.json → add icd_codes + omim

  OUTPUT JSON:
    {
      symptoms_extracted: string[],
      symptoms_with_hpo: [{term, hpo_code}],
      hpo_codes_used: string[],
      matches: [{
        orpha_code, name, confidence, gemini_confidence,
        hpo_score, combined_score, reasoning,
        icd_codes, omim, matched_hpo: [{code, name}]
      }]
    }

── VARIANT B: Gemini + Vertex AI Embeddings (original design) ──────────────────

  Step 1 — OCR (same as Variant A)
  Step 2 — Gemini: extract symptom list from text
  Step 3 — Vertex AI text-embedding-004: embed joined symptom string
  Step 4 — Cosine similarity against pre-built index.json
  Step 5 — Gemini: compare report diagnosis vs AI diagnosis
  Note: Requires pre-built index.json (generated by scripts/) and
        index.json is not committed to the repo.
        Also compares report_diagnosis vs ai_diagnosis with match type:
        "matches" | "differs" | "no_report_diagnosis"

6. LIVING BRIEF — /api/living-brief
================================================================================

  What: Generates a personalised clinical summary document for the patient
  Input: symptoms[], matches[] (top disease results)
  Process: Single Gemini call → structured markdown document
  Output:
    {
      brief: string (markdown),
      generatedAt: ISO timestamp,
      basedOn: { symptoms, topMatch: { name, orpha_code } }
    }

  Features of the generated brief:
    - Patient-readable explanation of the top diagnosis
    - Summary of matched symptoms
    - ICD and OMIM reference codes
    - Suggested next steps / questions for the doctor
    - Disclaimer that it is AI-generated and not a clinical opinion

  Model fallback: Same multi-model discovery as /api/diagnose
  (Fetches available Gemini models dynamically, tries each on 429 rate limit)

14. KEY DESIGN DECISIONS
================================================================================

  Why Next.js App Router?
    Server components + API routes in one codebase. No separate backend.
    API routes handle all GCP/AI calls server-side (API keys never exposed).

  Why not use Orphanet's own text descriptions?
    Orphanet XML exports do not include symptom descriptions in the parsed
    format we use. The alignments.json contains only codes and names.
    Clinical knowledge comes entirely from Gemini's training data + HPO.

  Why Gemini over OpenAI?
    Free tier availability, Google ecosystem alignment (same account as
    Vision + Vertex), and strong medical knowledge in Gemini 1.5/2.0.

  Why HPO + Gemini instead of pure LLM?
    Pure LLM matching is inconsistent — Gemini may not know rare diseases.
    HPO database provides evidence-based phenotype matching grounded in
    peer-reviewed clinical data (114,943 curated ORPHA-HPO associations).
    Combined scoring (55% LLM + 45% HPO) outperforms either alone.

  Why Jaccard similarity for HPO scoring?
    Jaccard = |intersection| / |union|
    Penalises diseases that have many HPO terms the patient doesn't have.
    More precise than raw overlap count.

  Why module-level caching for HPO data?
    phenotype.hpoa is 35 MB (282K rows). Parsing takes ~5s on first request.
    Module-level Maps persist across requests in the Next.js server process.
    After first request: lookup is O(1), no file I/O.

  Gemini model fallback strategy:
    /v1/models endpoint is polled once at startup to discover all available
    models that support generateContent. On 429 (rate limit), the next model
    in the list is tried automatically. Eliminates hardcoded model names
    and adapts to Google's model availability changes.
